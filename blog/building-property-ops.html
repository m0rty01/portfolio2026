Architecting PropertyOps: A deep dive

Published on March 10, 2026 • 8 min read

PropertyOps was born out of a simple realization: property management data is fragmented. Managers, workers, and tenants often use three different systems that don't talk to each other. This case study looks at how I built a unified platform to solve this.

The Tech Stack

I chose Next.js 15 for its robust routing and server-side capabilities. Coupled with Prisma ORM and PostgreSQL, it allowed for a type-safe data layer that scaled easily as I added more complex relations between units, tenants, and maintenance tickets.

Multi-Role Authorization

One of the biggest challenges was ensuring each user role only saw what they were supposed to. I implemented a robust RBAC (Role-Based Access Control) system that manages permissions at both the API level and the UI component level.

// Prisma schema snippet for multi-role
enum Role {
  MANAGER
  WORKER
  TENANT
}

model User {
  id    String @id @default(cuid())
  email String @unique
  role  Role   @default(TENANT)
}

Visualizing Maintenance Trends

Data is useless if you can't read it. I integrated Recharts to provide managers with live analytics on ticket resolution times, category breakdowns (plumbing vs. electrical), and worker efficiency. This transformed PropertyOps from a simple tracker into a decision-making tool.

Building PropertyOps taught me the importance of a clean data model before even touching the UI. When the foundation is solid, building features becomes a breeze.
